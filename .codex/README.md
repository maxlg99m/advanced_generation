# .codex

Эта папка содержит no-code агентскую инфраструктуру для генерации QA чек-листов.

Содержимое:

- `orchestrator/` — master-agent, основная точка входа для команд.
- `commands/` — command-wrapper файлы для запуска этапов.
- `prompts/` — системные prompts и prompts агентов.
- `rules/` — обязательные правила маршрутизации, покрытия, CSV и валидации.
- `skills/` — platform-specific skills для web и mobile.

Rules layer:

- `rules/agent-system_rules/` — правила агентского пайплайна: routing, inputs, handoff, coverage, traceability, validation, final report, logs и workspace cleanup.
- `rules/testrail_rules/` — правила CSV/TestRail: CSV contract, Section, Priority, Scenario, Platform, Labels, Title и Expected Result.

## 🔄 Цикл работы агентской инфраструктуры

Инфраструктура работает через три основные команды. Каждая команда делает только свой этап и не перескакивает к следующему без явного запроса пользователя.

### 1. `/init-qa-context` — подготовка контекста

Эта команда ничего не генерирует и не изменяет файлы. Она нужна, чтобы агент понял проект и проверил готовность входных данных.

Что происходит:

1. Агент читает `README.md`, `.codex/README.md` и `orchestrator/master-agent.md`.
2. Загружает правила из:
   - `rules/agent-system_rules/`;
   - `rules/testrail_rules/`.
3. Загружает prompts:
   - системный prompt;
   - prompts агентов пайплайна.
4. Проверяет skills для web/mobile.
5. Проверяет входные папки:
   - `data/input/requirements/`;
   - `data/input/design/`.
6. Определяет режим работы:
   - только ТЗ;
   - только дизайн;
   - ТЗ + дизайн;
   - нет входных данных.
7. Определяет платформу: web, mobile, web+mobile или unknown.
8. Сообщает, готов ли проект к `/generate-qa-checklists`.

Итог: пользователь получает короткий отчет о найденных файлах, режиме, платформе и готовности к генерации.

### 2. `/generate-qa-checklists` — генерация чек-листов

Это основная команда. Она запускает полный пайплайн: от чтения входных материалов до финального CSV для TestRail.

Перед стартом агент проверяет:

1. Был ли ранее выполнен `/init-qa-context`.
2. Есть ли папки `data/input/requirements/` и `data/input/design/`.
3. Есть ли хотя бы один входной файл: ТЗ или PNG/JPEG дизайн.
4. Нет ли неподдерживаемых дизайн-файлов.
5. Есть ли целевые папки `data/output/csv/` и `logs/`.

Дальше пайплайн идет по шагам:

1. **Ingestion Agent**  
   Проверяет входные файлы, форматы и режим работы. Если входы некорректны, пайплайн останавливается.

2. **Analysis Agent**  
   Анализирует ТЗ и/или дизайн. Из ТЗ выделяет требования, ограничения и тестируемые утверждения. Из дизайна собирает `ui_inventory`: экраны, кнопки, поля, таблицы, состояния и другие значимые UI-элементы.

3. **Checklist Generation Agent**  
   Создает атомарные чек-листы. Каждый чек-лист получает `Section`, `Title`, `Priority`, `Platform`, `Labels`, `Scenario` и `Expected Result`. Для дизайна проверки строятся по `ui_inventory`, а не по общему впечатлению от картинки.

4. **Coverage Review Agent**  
   Проверяет полноту покрытия. Смотрит, покрыты ли требования и обязательные UI-элементы, нет ли слабых названий, пустых полей, дублей и нарушений правил.

5. **CSV Export Agent**  
   Перед экспортом проверяет CSV-контракт, качество `Title`, обязательные поля и design coverage. Если блокеров нет, сохраняет финальный CSV в:

   ```text
   data/output/csv/final_checklists.csv
   ```

6. **Логи**  
   После генерации агент сохраняет только разрешенные markdown-логи:

   ```text
   logs/Questions.md
   logs/Mistakes.md
   logs/Details_and_Improve.md
   ```

Итог: пользователь получает CSV для TestRail и краткий отчет о режиме, платформе, количестве чек-листов, статусе review и возможных ограничениях.

### 3. `/clear-qa-workspace` — очистка рабочей области

Эта команда очищает только пользовательские и сгенерированные данные. Инфраструктурные файлы не удаляются.

Что может быть очищено:

- `data/input/requirements/`;
- `data/input/design/`;
- `data/output/`;
- `logs/`.

Что никогда не удаляется:

- `.codex/`;
- `.git/`;
- `README.md`;
- `.gitignore`;
- prompts;
- rules;
- skills.

Алгоритм безопасной очистки:

1. Агент перечисляет папки, которые будут очищены.
2. Сообщает, что `.codex`, Git-настройки и документация не будут затронуты.
3. Запрашивает явное подтверждение пользователя.
4. Если подтверждения нет, ничего не удаляет.
5. Если подтверждение есть, удаляет только содержимое разрешенных папок и сохраняет сами директории.

Итог: workspace очищен от входных и сгенерированных данных, но агентская инфраструктура остается на месте.
